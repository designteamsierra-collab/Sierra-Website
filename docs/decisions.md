## 2026-05-05 · Phase 0 recon strategy: REST API + yoast_head_json over HTML scraping

**Context:** Need to capture title/description/canonical/OG/Twitter/JSON-LD for every URL on the WordPress site. HTML scraping was the obvious approach.

**Decision:** Pull metadata via WordPress REST API's `yoast_head_json` field instead of scraping rendered HTML.

**Why:**

- 6 paginated REST calls covers all 129 records vs. 113 individual HTML fetches
- Returns structured JSON (no regex on HTML head)
- Bypassed Cloudflare WAF rules that blocked 9 specific URL paths from curl
- Yoast JSON-LD `@graph` arrives ready-to-mirror in Phase 2 — no JSON-from-HTML extraction needed

**Tradeoff:** Body-derived fields (H1, word count, image count, internal links) still require parsing the rendered HTML — those are unavoidable since REST `content.rendered` is Elementor-shortcoded mush. So we use REST for metadata, HTML for body. Two-source merge keyed by canonical URL.

## 2026-05-05 · Sitemap source of truth: regenerate from canonical URLs, not legacy sitemap

**Context:** The published `sitemap.xml` was generated by xml-sitemaps.com (third-party scraper), not Yoast. It contains 16 stale entries for leadership-team pages that 301-redirect to CPT canonicals (`/leadership-team/<slug>/`).

**Decision (proposed; awaiting user confirmation):** New site's `app/sitemap.ts` will generate from the 129 canonical URLs in `recon.json`, **not** mirror the existing sitemap entry-for-entry. Legacy stale URLs stay alive only as 301 redirects in `next.config.js`.

**Why:** Google has consolidated equity at the canonical URLs anyway. Submitting a sitemap that points at 301s would be a cleanup signal but adds noise. The 301s themselves preserve any backlinks that still target the old URLs.

## 2026-05-05 · Three SEO-upgrade redirects (user-approved)

**Context:** Live WP today 301s three legacy leadership URLs to `/page-not-found/`. The brief says preserve current behavior unless user explicitly approves changes.

**Decision (user-confirmed 2026-05-05):** Override the live-WP 404 behavior:

- `/senthil-kumar-ceo/` → `/leadership-team/senthil-kumar/` (same person; CPT migration renamed the slug — clear SEO upgrade)
- `/jayakanth-loganathan/` → `/leadership-team/` (person no longer at company — listing page preserves more equity than a 404)
- `/sooriya-sudhakar/` → `/leadership-team/` (same situation)

**Why:** Each of these legacy URLs likely retains some backlink equity. Routing them to `/page-not-found/` (which itself has noindex) destroys that equity. Routing the renamed-slug case to its successor preserves it cleanly; routing the departed-people cases to the listing keeps the equity within the leadership cluster of the site.

## 2026-05-06 · Phase 4: lead-capture gate replaces generic Clerk signup

**Context:** User reframed Phase 4 — instead of generic self-signup, the cockpit should be a **gated lead-generation experience**. Visitors must complete a qualified business-context form (full name, company email, company name, job title, website, use case, optional phone, consent) before they see any cockpit content. Personal/disposable email domains rejected. Lead must be captured in CRM **before** access is granted.

**Architecture decisions:**

1. **Three-layer email qualification** — domain blocklist (curated list of personal mail providers + disposable services in `lib/email-validation.ts`) + DNS MX-record validation + Clerk magic-link ownership verification. All three must pass.

2. **CRM-first transactional ordering** in `/api/lead/cockpit`: validate → MX check → CRM upsert → only then create Clerk user. If CRM fails, no Clerk user is created — keeps the two stores consistent.

3. **CRM stub via JSON-Lines (`lib/crm/jsonl.ts`)** for Phase 4. Identical `CrmClient` interface as the HubSpot client. When `HUBSPOT_PORTAL_ID` + `HUBSPOT_PRIVATE_APP_TOKEN` env vars are set (Phase 6), the resolver swaps in the HubSpot impl. Form behavior to the user is identical.

4. **Cockpit content via auth-gated iframe (Phase 4.B will port to React).** A faithful 4-simulator React port of the 2200-line prototype is its own focused chunk. For Phase 4 we ship a working gate end-to-end, with the cockpit page rendering an `<iframe src="/rise-cockpit/preview">` whose route handler streams the prototype HTML after re-checking auth. Prototype content is 100% interactive; nothing is lost. Port to native React deferred to Phase 4.B.

5. **Sign-in page hosts Clerk's `<SignIn>` component on our domain** (`/sign-in/[[...sign-in]]/page.tsx`) rather than redirecting to Clerk's hosted UI — keeps the Sierra visual chrome consistent.

6. **Middleware redirects unauthenticated `/rise-cockpit` to `/rise-cockpit/request-access`** (the form), NOT to `/sign-in`. First-time visitors should fill the qualification form, not see a sign-in screen for an account they don't have. Returning visitors click "Sign in" link below the form.

**Env vars added:** `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY`, `CLERK_SECRET_KEY`, `CLERK_WEBHOOK_SECRET`. Documented in `.env.local.example`. Without these, `pnpm build` still succeeds (Clerk SDKs warn but don't crash), but runtime auth fails — see `.env.local.example` for the full Clerk dashboard configuration checklist.

## 2026-05-05 · Node 24 LTS + corepack-managed pnpm

**Context:** Phase 1 needs a Node toolchain. Brief specified pnpm. Machine had no Node installed. winget's `OpenJS.NodeJS.LTS` resolves to Node 24.15.0 today (Node 24 went LTS in late 2025).

**Decision:** Install Node 24 LTS via `winget install OpenJS.NodeJS.LTS`. Enable pnpm via `corepack` rather than installing pnpm separately — locks the pnpm version per-project via `package.json`'s `packageManager` field, which is the modern Node-recommended approach and what Vercel honors.
